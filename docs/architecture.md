# Architecture

> **OpenTone** — offline-first personal music library and desktop player for music you own.

---

## Overview

OpenTone is a **Tauri v2 desktop application** with a React frontend and a Rust backend. The frontend provides the user interface (library views, playback controls, settings), while the Rust backend handles file system operations, metadata extraction, SQLite persistence, and artwork processing.

**OpenTone is a desktop app, not a hosted web service.**
- **Development access:** `npm run tauri dev`
- **Production access:** Desktop installer/app bundle generated by `npm run tauri build`
- **No Vercel deployment for the core app.** A marketing/docs/download website could optionally be deployed to Vercel, but the actual application is a desktop native binary.

---

## Stack

| Layer | Technology |
|-------|-----------|
| Desktop shell | [Tauri v2](https://v2.tauri.app/) |
| Frontend | React 19 + TypeScript |
| Build tool | Vite 6 |
| Styling | Tailwind CSS 4 (calm color scheme) |
| Local database | SQLite via `rusqlite` (bundled, no system dep) |
| Metadata | [Lofty](https://crates.io/crates/lofty) — ID3, Vorbis Comments, MP4 |
| Artwork | `image` crate — PNG/JPEG decode, Lanczos3 resize |
| Audio | Browser `Audio()` API (Phase 1); Tauri asset protocol planned (Phase 2) |
| File scanning | `walkdir` crate |

## Data Flow

### Library Import
```
User clicks "Import"
  → open() dialog (Tauri plugin)
  → scan_folder(path) Tauri command
    → walkdir::WalkDir recursive scan
    → is_supported() extension filter
    → metadata::extract_metadata() via Lofty
    → db::upsert_track() SQLite insert/update
  → get_library() / get_albums() / get_artists()
  → Frontend re-renders
```

### Artwork Loading
```
Track row rendered → loadArtwork(track)
  → get_album_artwork(track_id) Tauri command
    → db::get_track_by_id() → get track path
    → artwork::extract_and_cache_artwork()
      → Lofty reads embedded picture
      → image::load_from_memory() decode
      → resize 256×256 Lanczos3 if needed
      → save PNG to cache dir
    → artwork::cached_artwork_as_data_uri()
      → read PNG → base64 → data:image/png;base64,...
  → Frontend displays <img src={dataUri}>
```

### Playback
```
User clicks track → handlePlayTrack()
  → add to queue at queueIndex
  → useEffect sets audio.src = track.path
  → Browser Audio() API plays file from local filesystem
  → timeupdate / ended events → UI updates
```

### Playlist Management
```
User creates playlist → create_playlist(name) Tauri command
  → db::create_playlist() → INSERT into playlists table
User adds track → add_to_playlist(playlist_id, track_id)
  → db::add_to_playlist() → INSERT into playlist_tracks + update updated_at
User views playlist → get_playlist_tracks(playlist_id)
  → JOIN tracks + playlist_tracks → ORDER BY position
```

## Database Schema

8 tables, all created in `db::initialize_db()`:

| Table | Purpose | Key Columns |
|-------|---------|-------------|
| `tracks` | Core music library | id, path (UNIQUE), title, artist, album, duration, is_favorite, has_artwork |
| `albums` | Aggregated album view | id, title, artist, year |
| `artists` | Aggregated artist view | id, name (UNIQUE) |
| `playlists` | User-created playlists | id, name, created_at, updated_at |
| `playlist_tracks` | Track membership in playlists | playlist_id, track_id, position |
| `indexed_paths` | Imported directory tracking | path (UNIQUE), last_indexed_at, file_count |
| `playback_history` | Listening history (reserved) | track_id, played_at |
| `settings` | Key-value preferences | key (PK), value |

## Design Principles

- **Calm** — Minimal, warm, restrained interface. No flashing UI, no algorithmic feeds.
- **Offline-first** — No account required. No data leaves your machine.
- **Ownership-first** — You own your music files. OpenTone just helps you organize and play them.
- **Transparent** — No recommendations, no social features, no tracking.
- **Privacy-respecting** — Zero telemetry, zero data collection, zero cloud dependencies.
